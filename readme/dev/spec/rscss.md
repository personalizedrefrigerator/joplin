# RSCSS quick reference (desktop app)

This is a practical cheat sheet for styling the desktop app with [RSCSS](https://ricostacruz.com/rscss/) and SCSS. For the bigger picture of how stylesheets are built and included, read [desktop_styling.md](./desktop_styling.md) first.

RSCSS is not a framework — it's a small set of naming conventions that keep CSS predictable as the app grows. The whole idea is: **target exactly the element you mean, and nothing else.**

## The core rules

1. **Components are two or more words, dash-separated.** A component is a reusable, self-contained block. Its class is the entry point for all its styles.

	```scss
	.share-folder-dialog { /* ... */ }
	.search-form { /* ... */ }
	```

2. **Elements are a single word.** The pieces inside a component get short, one-word classes. If you really need two words, concatenate them (no dash) so they stay visually distinct from components.

	```scss
	.search-form {
		> .field { /* ... */ }
		> .action { /* ... */ }
		> .firstname { /* two words, but concatenated */ }
	}
	```

3. **Always use the `>` child selector.** This is the rule that makes RSCSS work — it stops a component's styles from "bleeding" into nested components that happen to reuse a name.

	```scss
	.article-card {
		.title    { /* ✗ leaks into any nested .title */ }
		> .author { /* ✓ only this card's direct author */ }
	}
	```

4. **Variants are dash-prefixed.** A variant is a state or modifier toggled with an extra class (`-active`, `-disabled`, `-small`). The dash prefix keeps variants unambiguous from elements and reads like a flag.

	```scss
	.like-button {
		&.-wide { /* ... */ }
		&.-disabled { /* ... */ }
	}

	.share-recipient {
		> .title { /* ... */ }
		> .title.-small { /* element variant */ }
	}
	```

5. **Prefer class names over tag selectors.** `> .name` is clearer and more robust than `> h3` — the markup can change without breaking the style.

6. **Keep nesting shallow.** Limit selectors to one `>` level. Reassess anything two levels deep, and avoid three. When a block has its own elements, promote it to its own component rather than chaining (`.share-recipient`, not `.recipients > .recipient-list > .recipient`).

## Joplin-specific conventions

These layer on top of plain RSCSS:

- **One stylesheet per component.** RSCSS keeps each component in its own file (see [css-structure.md](https://github.com/rstacruz/rscss/blob/main/docs/css-structure.md)). A simple component lives next to its code as `MyComponent/style.scss`. A component built from several sub-components uses a `MyComponent/styles/` folder with one file per component, plus a `MyComponent/style.scss` that `@use`s each of them — see [Sidebar/styles/](https://github.com/laurent22/joplin/tree/dev/packages/app-desktop/gui/Sidebar/styles) and [ShareFolderDialog/styles/](https://github.com/laurent22/joplin/tree/dev/packages/app-desktop/gui/ShareFolderDialog/styles).

- **Register the stylesheet** with an `@use` line so it gets compiled into the bundle:
	- Component-local styles → add to the root [packages/app-desktop/style.scss](https://github.com/laurent22/joplin/blob/dev/packages/app-desktop/style.scss).
	- App-wide / reusable styles → put the file in [gui/styles/](https://github.com/laurent22/joplin/tree/dev/packages/app-desktop/gui/styles) and register it in [gui/styles/index.scss](https://github.com/laurent22/joplin/blob/dev/packages/app-desktop/gui/styles/index.scss). `link-button.scss` is a good example of a small shared style.

- **Use theme CSS variables, not JS theme lookups.** Every key on the theme object is exposed as a `--joplin-*` CSS variable (generated by [themeToCss.ts](https://github.com/laurent22/joplin/blob/dev/packages/lib/services/style/themeToCss.ts)). The name is the camelCase key converted to dash-case:

	| Theme key (JS) | CSS variable |
	| --- | --- |
	| `theme.color` | `var(--joplin-color)` |
	| `theme.backgroundColor` | `var(--joplin-background-color)` |
	| `theme.dividerColor` | `var(--joplin-divider-color)` |
	| `theme.fontSize` | `var(--joplin-font-size)` |
	| `theme.mainPadding` | `var(--joplin-main-padding)` |

	Numeric sizes are emitted with a `px` unit already, so use the variable directly (`font-size: var(--joplin-font-size);`). For arithmetic, use `calc()`: `margin-bottom: calc(var(--joplin-main-padding) / 2);`.

- **Express dynamic styles with variants or `:nth-child`, not props.** Where styled-components passed a prop (`index`, `selected`, `disabled`), reach for a CSS feature instead: a dash-variant class toggled in the component, or a structural selector like `&:nth-child(even)`. This keeps the styling in the stylesheet.

- **Indent with tabs**, single-quote any strings, and only comment when the *why* is non-obvious (per [coding_style.md](../coding_style.md)).

## Worked example

[ShareFolderDialog](https://github.com/laurent22/joplin/tree/dev/packages/app-desktop/gui/ShareFolderDialog) was the first component migrated from styled-components and is a representative reference for the patterns above. Note how the recipient row is its own `.share-recipient` component rather than a deep `.share-folder-dialog > .recipients > .recipient-list > .recipient` chain — every selector stays at most one `>` deep.

```scss
.share-folder-dialog {
	min-width: 500px;

	> .folder {
		display: flex;
		align-items: center;
		// `gap` spaces the icon from the label, so no nested `> .icon` rule is needed.
		gap: 8px;
		border: 1px solid var(--joplin-divider-color);
	}
}

.share-recipient {
	// Alternating row colours, previously passed as an `index` prop.
	background-color: var(--joplin-background-color);
	&:nth-child(even) {
		background-color: var(--joplin-odd-background-color);
	}

	> .name { flex: 1; }
	> .status-icon { margin-right: .6em; }
}
```

## Common pitfalls

- **Styling a nested component by its component class.** `.order-form > .slider-box { }` is wrong — `slider-box` is its own component. Give it an extra one-word element class for layout (`<div class="slider-box quantityslider">`) and target `.order-form > .quantityslider`.
- **Descendant selectors by habit.** Reaching for `.dialog .button` instead of `> .button` is the most common way styles leak. Default to `>`.
- **Deep nesting.** If selectors are getting more than a couple of `>` levels deep, that's usually a sign the inner block should be its own component with its own stylesheet.
- **Generic single classes.** `.active`, `.title`, `.button` on their own are too broad. They only work scoped under a component (`.like-button.-active`).
